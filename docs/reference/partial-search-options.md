# 부분명(짧은 이름) 검색 해결 방안 조사

조사일: 2026-07-08 · 대상 API: data.go.kr `tn_pubr_public_nutri_process_info_api` (dataset 15100066)
조사 방식: 발급된 서비스키로 **라이브 API 직접 호출 검증** + data.go.kr / foodsafetykorea 명세 조회

---

## 0. 핵심 결론 (요약)

- **이 API는 부분/와일드카드/LIKE 검색을 지원하지 않는다.** `foodNm`은 **완전일치(exact full-string)** 전용임을 라이브로 확정.
- **최선책 = 방식 2 (전량 로컬 미러 + 기존 FTS5 부분검색).** 전량이 **591 콜**이라 **개발키 일일한도(≈1,000콜) 안에서 하루면 끝난다.** "며칠에 걸친 소량 배치"라는 전제는 불필요.
- 이 결론은 기존 `data-pipeline-spec`의 **월 1회 전체 스냅샷 교체** 전략과 이미 일치한다. 온디맨드 완전일치 조회는 "월간 미러 사이에 새로 등록된 신상품" 보완용으로만 남기면 된다.

## 1. 라이브 검증 증거 (완전일치 전용)

동일 서비스키로 `https://api.data.go.kr/openapi/tn_pubr_public_nutri_process_info_api` 직접 호출(type=json, UTF-8 인코딩):

| 쿼리 `foodNm` | 성격 | resultCode | totalCount |
|---|---|---|---|
| `코카콜라` | 완전일치 | 00 | **6** (반환명 전부 정확히 `코카콜라`) |
| `새우깡` | 완전일치 | 00 | **6** (전부 `새우깡`) |
| `초코무스` | 완전일치 | 00 | **10** (전부 `초코무스`) |
| `떠먹는초코무스케이크` | 완전일치(실데이터 원문) | 00 | **2** |
| `떠먹는초코무스` | **접두(prefix)** | 03 NODATA | 0 |
| `코카` | 부분(prefix) | 03 NODATA | 0 |
| `코카콜라제로` | 부분/변형 | 03 NODATA | 0 |
| `환타` / `포카칩` / `꼬북칩` | 짧은 통칭 | 03 NODATA | 0 |
| `like_foodNm=콜라` | LIKE 파라미터 시도 | 10 INVALID_PARAM | — |

**판정:** `떠먹는초코무스`(실제 등록명 `떠먹는초코무스케이크`의 접두)조차 0건 → **접두도 부분도 아닌 순수 완전일치**. `초코무스=10`은 부분매칭이 아니라 "정확히 `초코무스`로 등록된 제품 10건"이며, 반환된 `foodNm`이 전부 쿼리와 문자 단위로 동일함으로 확인됨. LIKE류 파라미터는 `INVALID_REQUEST_PARAMETER_ERROR`(code 10).

> tn_pubr_* 표준데이터 API는 자동생성 계열로 **컬럼=값 완전일치 필터만** 제공. 서버측 부분검색 기능이 스펙에 없음(라이브로 재확인).

## 2. 전량/증분 미러 비용 (라이브 측정)

| 항목 | 측정값 | 출처 |
|---|---|---|
| totalCount(전체 행) | **590,542** | 라이브 |
| numOfRows 최대 | **1,000** (2000은 즉시 실패) | 라이브 |
| 전량 1회 통과 콜 수 | **ceil(590542/1000) = 591콜** | 계산 |
| 1콜 응답시간(numOfRows=1000) | **≈70초** (행이 무거움) | 라이브 |
| 개발키 일일 한도 | ≈**1,000 콜/일** | 포털 기본값 |
| 운영키 일일 한도 | ≈**100,000 콜/일** | 포털 기본값 |

**일수 산정:**
- **개발키(1,000콜/일): 591콜 → 하루(한 번의 일일한도) 안에 전량 완료.** 콜 수 기준 여유 있음.
- 운영키(100,000콜/일): 591콜은 무시할 수준, 여러 번 반복도 가능.
- **실제 병목은 일일 콜 한도가 아니라 벽시계 시간**: numOfRows=1000일 때 콜당 ~70초라 순차 591콜이면 ~11.5시간. numOfRows를 200으로 낮추면 콜당 빨라지지만 2,953콜이 되어 개발키 한도를 넘음(운영키면 무관). → 야간 1회 배치로 처리하면 충분.
- **재수집 주기**: 원천이 월 병합 갱신이므로 **월 1회** 전량 재수집이면 됨 → 어떤 한도로도 여유. "매일 반복 다운로드" 우려는 이 유스케이스에 해당하지 않음.

## 3. 대체 소스 부분검색 (방식 5)

- **식약처 식품영양성분DB API** — foodsafetykorea `I2790`(~2023, 갱신 중단) / data.go.kr **15127578**(현행) / foodsafetykorea `I0750·I0680`.
  - 검색 파라미터 **`DESC_KOR`(품목명)** 가 **부분일치(LIKE) 검색을 지원**함(다수 활용사례에서 "하리보" 등 부분어로 다건 반환 확인 — 문서/블로그 기준). ⚠️ **본 조사에서 라이브 미검증**: 해당 API는 별도 포털(식약처 데이터포털)의 별도 keyId가 필요하며 현재 NutriRank 키로는 호출 불가.
  - **치명적 한계(품질):** 이 성분DB는 **대표품목 중심의 영양성분 DB(수만 건)** 로, NutriRank가 등급화하는 **15100066의 브랜드 SKU(59만 건, 품목제조보고번호·제조사 포함)와 입도/식별자가 다르다.** 즉 여기서 얻은 이름이 15100066의 `foodNm`과 1:1로 맞지 않아, "부분검색으로 후보명 얻기 → 15100066에 완전일치 재조회" 파이프라인이 **이름 불일치로 자주 실패**함. → 자동완성 힌트 정도의 보조 용도로만 유효.

## 5. 권고 순위

1. **[채택] 방식 2 — 전량 로컬 미러 + 기존 FTS5 트라이그램 부분검색.**
   근거: (a) 591콜로 **개발키 하루 안에 전량 확보** 가능하고 실제 병목은 벽시계 시간뿐이라 야간 1회 배치로 해결; (b) 이미 보유한 **FTS5 퍼지검색 + 월간 전체 스냅샷 교체 파이프라인**을 그대로 쓰므로 신규 구현이 거의 없음; (c) `꼬북칩·환타·포카칩` 등 모든 통칭이 **로컬 부분검색으로 완전 커버**되어 API 완전일치 한계를 원천 제거. → **온디맨드 완전일치 조회는 "월간 미러 사이 신상품" 폴백으로 격하.**
2. **[보완] 방식 4 — 큐레이션 시드.** 전량 미러 완주 전 부트스트랩 및 인기순 랭킹 보강용 임시 수단.
3. **[보완] 방식 3 — 쿼리 정규화.** 로컬 FTS 위에 얹는 UX 향상층(맛·향·용량 접미 제거로 재현율↑). 단독 해법 아님.
4. **[선택] 방식 5 — 성분DB 자동완성.** 필요 시 검색어 추천 힌트로만. 등급 데이터 소스로는 부적합.
5. **방식 1**은 불가.

## 6. 미검증/주의 사항 (Flag)

- 방식 5의 `DESC_KOR` 부분검색은 **문서/사례 기반**이며 본 조사에서 라이브 미검증(별도 포털 keyId 필요). 채택 전 실호출 확인 필요.
- 개발/운영키 일일 한도(1,000 / 100,000)는 **포털 기본값**이며, **실제 승인 한도는 마이페이지 활용신청 상세에서 확정**해야 함.
- numOfRows=1000 콜당 ~70초는 1회 측정치(콜드스타트 포함 가능). 대량 배치 시 페이지 크기·동시성 튜닝 필요.
