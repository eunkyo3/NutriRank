# 분석 질문 정의 — 가설·검증·서빙 화면 매핑

작성 2026-07-24 · 근거 데이터: 전량 재적재 스냅샷(2026-07-23 적재, 제품 70,013건 · 등급 산출 69,989건 · 9카테고리 · `nutriscore-2023-v1`)

이 문서는 프로젝트의 목표 정의와 가설 체계를 박제한다. 각 가설은 **데이터 전체로 참/거짓을 판정할 수 있는 문장**이어야 하며,
검증 수치는 화면이 아니라 DB 실측에서 나온다(수치 산출 쿼리는 각 절에 명시).

## 1. 목표 정의 (2026-07-23 확정)

| 층 | 목표 |
|---|---|
| **서비스 목표** | 소비자가 간식·음료를 고를 때 **같은 카테고리 안에서 더 건강한 선택**을 하게 돕는다. |
| **분석 목표** | 가설 H1~H4를 전수 데이터로 검증하고, 그 결과를 화면으로 서빙한다. |

**발표 서사:** 문제 정의(H1) → 그래서 선택이 중요(H2·H3) → 어떻게 고르나(H4 + 등급 근거)
→ 데모("내가 먹는 ○○은 어디쯤인가": 검색→상세) → 이를 가능하게 한 12단계 파이프라인.

> 주의: "내 제품이 카테고리 어디쯤인가"는 가설이 아니라 **제품 기능이자 데모 시나리오**다.

## 2. 가설 체계

### H1. 시판 가공 간식·음료의 대부분은 영양 기준 하위 등급이다 — **검증됨**

- **실측:** D+E **75.9%** (E 57.5% · D 18.5% · C 13.9% · B 9.0% · A 1.2%), 등급 산출 69,989건 기준
  - 구 스냅샷(48,215건 · 6카테고리)에서는 79.2%였다 — 카테고리 확장(차음료·아이스크림 등 상대적으로 나은 군 포함) 후에도 결론 불변
- **역할:** "문제가 실재한다"는 문제 정의의 근거
- **화면:** 홈 등급 분포 (DB 계산값이므로 재적재 시 자동 갱신)
- **쿼리:** `select health_grade, count(*) from grade_result where gradable=1 group by health_grade`

### H2. 무엇을(어떤 카테고리를) 먹느냐가 1차 결정 요인이다 — **검증됨**

- **실측:** 카테고리 평균 건강 점수(낮을수록 건강)가 최대 **6.9배** 차이:

  | 카테고리 | 평균 점수 | D·E 비율 |
  |---|---:|---:|
  | 초콜릿 | 26.3 | 98.1% |
  | 비스킷 | 22.0 | 95.3% |
  | 스낵/칩 | 17.0 | 80.5% |
  | 캔디/젤리 | 15.6 | 75.4% |
  | 차음료 | 10.9 | 61.6% |
  | 주스 | 10.4 | 75.1% |
  | 아이스크림/빙과 | 9.2 | 35.7% |
  | 탄산음료 | 6.9 | 48.8% |
  | 커피음료 | 3.8 | 23.7% |

- **화면:** /analytics 카테고리 비교 (평균 점수 막대 + 지표 대조표)
- **쿼리:** `category_agg_snapshot` 최신 행 (사전계산)

### H3. 같은 카테고리 안에서도 격차가 커서, 제품 선택만으로 더 건강해질 수 있다 — **검증됨**

**서비스의 존재 이유를 정당화하는 핵심 가설** — 같은 카테고리 제품이 다 비슷하면 순위 서비스는 무의미하다.

- **실측:** 모든 카테고리에서 점수 범위가 40점 안팎으로 벌어진다:
  초콜릿 **−7 ~ +42**, 스낵/칩 −6 ~ +45, 비스킷 −6 ~ +44, 캔디/젤리 −8 ~ +41,
  탄산음료 −6 ~ +36, 주스 −4 ~ +36, 차음료 −6 ~ +32, 아이스크림 −4 ~ +33, 커피음료 −6 ~ +31.
  즉 "최악의 카테고리"인 초콜릿 안에도 A급 점수대 제품이 존재한다 — 카테고리를 바꾸지 않아도 선택으로 개선 가능.
- **화면:** /analytics 대조표의 "점수 범위" 열 + 순위 화면 상단 범위 요약 한 줄
- **쿼리:** `select category_id, min(health_score), max(health_score) from category_ranking group by category_id`

### H4. 등급을 깎는 주범 성분은 카테고리마다 다르다 — **검증됨**

- **실측:** 성분↔점수 피어슨 상관계수(양수 = 많을수록 덜 건강):

  | 카테고리 | 당류 | 나트륨 | 포화지방 | 에너지 | 주범(에너지 제외 최대) |
  |---|---:|---:|---:|---:|---|
  | 탄산음료 | **0.868** | 0.128 | 0.045 | 0.845 | 당류 |
  | 차음료 | **0.804** | 0.128 | 0.062 | 0.836 | 당류 |
  | 커피음료 | **0.816** | 0.570 | 0.613 | 0.678 | 당류 |
  | 주스 | **0.732** | 0.190 | 0.197 | 0.631 | 당류 |
  | 캔디/젤리 | **0.820** | 0.308 | 0.429 | 0.788 | 당류 |
  | 초콜릿 | **0.795** | 0.179 | 0.379 | 0.597 | 당류 |
  | 스낵/칩 | 0.458 | 0.515 | **0.687** | 0.562 | 포화지방 |
  | 비스킷 | 0.496 | 0.203 | **0.579** | 0.556 | 포화지방 |
  | 아이스크림/빙과 | 0.640 | 0.384 | **0.823** | 0.908 | 포화지방 |

- **인사이트:** 음료·당과류는 **당류**, 스낵/칩·비스킷·아이스크림은 **포화지방**이 등급을 깎는 1차 성분이다.
  (에너지는 다른 성분들의 합산 결과라 대부분 카테고리에서 함께 높게 나온다 — 주범 판정에서는 원인 성분 3종을 우선한다.)
  같은 "간식"이라도 피해야 할 성분이 다르므로, 카테고리별 대시보드가 성분 단위 근거를 보여줄 가치가 있다.
- **화면:** /analytics/[categoryId] 4성분 상관 표 + 당류 산점도 + 주범 인사이트 문장

### H5. 신제품일수록 건강한가 — **기각 (2026-07-23)**

**기각 사유 — 데이터생성일자(`crtYmd` → `product.data_gen_date`)는 출시일 대용으로 쓸 수 없다:**

1. **일괄 이관 흔적:** 구 스냅샷 실측에서 2021년 6,965건 전부가 2021-06-30 하루에 몰림.
   2020년(2,964건)·2022년(1,362건)·2019년(492건)도 사실상 단일 날짜. 제품 출시 이벤트가 아니라
   식약처 DB의 일괄 적재 날짜다. (2024년 이후는 자연 분산 — 최다 날짜 234건 수준)
2. **갱신일 혼입:** 파이프라인이 식품코드당 최신 데이터생성일자 행만 남기므로, 이 값은 "처음 등재된 날"이
   아니라 "마지막으로 갱신된 날"에 가깝다. 2015년 출시 제품도 작년에 성분이 갱신됐으면 최근 날짜로 찍힌다.
   "최근 날짜 = 신제품" 등식 자체가 불성립.

**발표 활용:** 기각 과정 자체가 재료다 — "가설 수립 → 날짜 필드 분포 실측 → 일괄 이관+갱신일 혼입 확인 →
검증 불가 판정 → 기각". 검증된 가설만큼 분석의 깊이를 보여준다.

> 대안 검토: 진짜 출시/등록일은 식약처 품목제조보고 계열 데이터셋 조인이 필요하나 식품코드 체계가 달라
> 매칭 비용이 커서 비추천. 원천 API의 데이터기준일자(#56) 필드도 미수집이며 역시 등록일이 아니다.

## 3. 화면 ↔ 가설 매핑 요약

| 화면 | 답하는 질문 | 가설 |
|---|---|---|
| 홈 (등급 분포) | 시판 간식·음료는 얼마나 건강한가? | H1 |
| /analytics (비교) | 무엇을 먹느냐는 얼마나 중요한가? | H2 |
| /rankings/[id] (순위) | 같은 카테고리 안에서 선택은 얼마나 중요한가? | H3 |
| /analytics/[id] (대시보드) | 이 카테고리의 등급을 깎는 주범은 무엇인가? | H4 |
| 검색 → 상세 | 내가 먹는 제품은 어디쯤인가? | (가설 아님 — 데모 시나리오) |
